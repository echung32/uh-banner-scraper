### Root-cause fix (landed): don't advance last_synced_at on partial/error

`syncTerm` calls `markTermSynced(..., status, ...)` on both the `partial` and `error` paths
([sync.ts:144](../../web/src/lib/ingest/sync.ts#L144), [:154](../../web/src/lib/ingest/sync.ts#L154)),
and `markTermSynced` previously stamped `last_synced_at` regardless of status — so a throttled
term looked complete to the read path (which gates the SQL branch on `last_synced_at IS NULL`).
That's the actual bug behind `202430`'s bad state.

Fixed in [upsert.ts](../../web/src/lib/db/upsert.ts) (`markTermSynced`): `last_synced_at` is
now only advanced when `status = 'ok'`
(`last_synced_at = CASE WHEN ? = 'ok' THEN ? ELSE last_synced_at END`). A never-synced
partial/error term stays `NULL` (re-queued for a clean retry / served by the dynamic page
cache), and a previously-`ok` term keeps its timestamp if a later re-sync goes partial.
`last_sync_status` + `section_count` are still recorded for observability. So the data reset
above is no longer needed for future throttled terms — they self-heal on the next sweep.
